# 用例编写规范

## 基本原则

- 用例必须面向真实业务场景，不能脱离当前项目的业务上下文。
- 用例步骤必须可以被测试人员直接执行，不能依赖测试人员自行猜测前置数据。
- 预期结果必须可以观察或验证，包含页面表现、数据状态或业务状态中的至少一项。
- 不允许使用"正常""正确""可用"等模糊表达作为唯一预期。
- 核心业务链路必须覆盖成功、失败、状态不允许、权限拦截、数据一致性、日志追溯和重复操作。
- 涉及审批、批准、生效、发布等关键动作时，按业务要求验证签名、审批意见和操作日志。
- 涉及上下游模块的数据引用时，必须验证上下游状态和版本一致。

---

## 用例粒度控制

生成用例时优先保证业务风险覆盖，不为低价值差异拆分过多用例。

### 应拆分为独立用例

- 正向主流程、关键异常流程和权限拦截。
- 会导致不同页面反馈、数据状态、业务状态或日志结果的场景。
- 不同用户角色、权限范围、数据状态会影响结果的场景。
- 跨模块联动、状态流转、消息 / 待办 / 通知、审计追踪等关键副作用。
- PRD 明确列为独立验收标准的字段、按钮、规则或流程。

### 应合并为一条用例

- 同一区域的字段、按钮、文案、默认状态和布局展示，合并为一条 `UI校验` 用例。
- 同一字段多种非法输入若拦截方式和提示文案相同，可合并为一条字段校验用例。
- 多个低风险文案或样式差异，除非 PRD 明确验收，否则合并到区域展示用例。
- 同一接口失败、网络失败、超时失败若页面处理一致，可合并为一条异常处理用例。
- 同一列表的排序、分页、筛选若 PRD 未要求逐项验收，可按常用组合覆盖。

### 不应生成

- 只验证"功能正常""页面正确"等不可判定结果的用例。
- 仅改变测试数据值但验证目标和预期完全相同的重复用例。
- 参考模板中存在但当前 PRD、知识库和用户补充均未涉及的场景。
- 当前项目不具备的能力场景；应在覆盖率对照表写不适用原因。

## 字段规范

文件级结构、需求问题清单、用例统计摘要和需求覆盖率对照表的写法以 `workflow_rules.md` 为准；本节只定义标准用例表的字段写法。

### 标准输出字段

输出 Markdown 表格必须使用"分组列 + 固定业务字段"的结构。默认表头为以下 12 列（3 个分组列 + 9 个通用字段）；如业务层级需要，可在 `三级分组` 后、`用例名称` 前连续增加 `四级分组`、`五级分组` 等分组列。**输出表格的分组列数必须等于本文件所有用例实际用到的最大分组深度**：仅当至少一条用例需要更深层级时才扩展分组列，若只有一级或二级业务层级，也允许后续分组值按用例实际层级留空。

> 若项目在 `business_constants.PROJECT_SPECIFIC_HEADERS` 中声明了追踪字段，表头会自动拼接到通用字段列后；具体追踪字段以本项目 `business_constants.py` 为准。

| 一级分组 | 二级分组 | 三级分组 | 用例名称 | 优先级 | 创建人 | 用例描述 | 前置条件 | 用例步骤 | 预期结果 | 备注 | 用例标签 |
|---|---|---|---|---|---|---|---|---|---|---|---|

### 分组字段

- `一级分组` 必须填写，对应用例在系统中的一级菜单或业务域。
- `二级分组`、`三级分组` 及后续分组列按当前用例适用层级填写。**某条用例的某级分组一旦有值，其所有更深层级分组列也必须有值**（即同一用例的分组深度必须填到底，不允许"三级分组有值、四级分组为空"等中间层空值）。某用例不适用更深层级时，该级及其更深层级整体留空，不得为了补齐层级写入"通用场景"等占位值。
- 当三级分组不足以表达业务层级时，可以继续增加 `四级分组`、`五级分组` 等分组列；新增分组列必须按层级连续出现，且位于 `用例名称` 前。例如：`一级分组 | 二级分组 | 三级分组 | 四级分组 | 用例名称 | ...`。
- **禁止保留冗余的空分组列**：表头中的分组列数必须等于本文件所有用例实际用到的最大分组深度。若某一级（含其更深层级）分组列在全部用例中均为空，则该列不得出现在表头中。新建文件按实际层级写表头；追加用例时若新用例需要更深层级，必须同步扩展表头并补齐已有用例对应层级。示例：
  - 合格：所有用例最深用到三级分组 → 表头为 `一级分组 | 二级分组 | 三级分组 | 用例名称 | ...`
  - 合格：至少一条用例用到四级分组，且无任何用例用到五级 → 表头为 `一级分组 | 二级分组 | 三级分组 | 四级分组 | 用例名称 | ...`
  - 不合格：所有用例最深只用到四级分组，但表头仍保留空的 `五级分组` 列
- **分组层级建议最多到四级**：生成用例时优先用一/二/三级分组承载自然业务层级；确需更细粒度时最多扩展到 `四级分组`，且仅当至少一条用例必须使用四级才能准确定位子功能、子页面或细分校验域时才扩展该列。**最多不超过 `五级分组`（代码硬限制，超过会报 `group_depth_exceeded` ERROR）**；超过四级的场景应优先重新组织分组方式（调整二级/三级命名、拆分为多个输出文件）；业务上确需 `五级分组` 的场景，必须在文件元信息块的 `生成假设` 中说明超出四级的具体原因。
- **任一分组至少 2 条用例才创建**：任何分组（二/三/四级等）下的用例数必须 ≥ 2；只有 1 条用例时不创建该层级分组，将用例归入上级分组或并入相邻同级分组，对应分组列留空。
- **某级分组只有唯一值时不创建该级**：同一上级分组下，若某级分组（三/四/五级等）只有一个唯一的分组值（即所有用例的该级分组名称完全相同），说明该级分组没有区分作用，应留空该级分组列，用例直接归入上级分组。
- 各分组层级的填写内容与示例如下（租户、组织、数据域等隔离信息如项目适用，应写入前置条件或分组，不体现在输出文件路径中）：

| 分组层级 | 填写内容 | 示例 |
|---|---|---|
| `一级分组` | 一级菜单或业务域 | `<TODO: 业务示例>` |
| `二级分组` | 下一层菜单、功能模块或影响范围 | `<TODO: 业务示例>` |
| `三级分组` | 二级菜单、页面、操作类型或子流程 | `<TODO: 业务示例>` |
| `四级分组`及后续层级 | 子功能、子页面、细分操作对象或更细的校验域 | `<TODO: 业务示例>` |
- 同一输出文件内，同一业务模块的分组命名必须保持一致。
- 相同完整分组路径的用例必须连续放在一起，不得被其他分组路径用例打断。
- **同一一级分组下的所有用例必须聚集在连续区间内**，不得被其他一级分组的用例打断；同一一级分组内按二级、三级分组业务流程顺序排列。
- 同级分组下，用例必须按优先级排序：`P0` 排最前面，`P1` 排在 `P0` 后面，`P2` 排最后；相同优先级内保持业务流程或生成顺序。

### 用例名称

- 用一句话说明本条用例的验证目标，20 字以内为佳。
- 格式建议：`[操作对象] + [动作] + [关键条件]`，示例：

| 写法 | 评级 | 说明 |
|---|---|---|
| <TODO: 合格示例，如"提交后状态变更为已生效"> | 合格 | — |
| <TODO: 合格示例，如"未生效时进入下一环节被拦截"> | 合格 | — |
| <TODO: 合格示例，如"文件格式不匹配时上传失败"> | 合格 | — |
| 测试某功能 | 不合格 | 过于宽泛 |
| 验证某模块 | 不合格 | 无法定位具体场景 |
- 按 UI 设计图、截图、原型图或界面交互说明生成的 `UI` 展示校验用例，`用例名称` 必须包含 `UI校验`，例如 `配置区UI校验`、`结果区UI校验`、`弹窗UI校验`。
- 同一输出文件的同一分组内用例名称不允许重复；不同需求拆分到不同 Markdown 文件时，允许复用通用场景名称。

### 优先级

- 只能使用 `P0`、`P1`、`P2`，遵守 `priority_rules.md` 的判断规则。

### 前置条件

- 必须写清所有外部依赖：

| 依赖类型 | 必填内容 |
|---|---|
| 用户类型与账号状态 | <TODO: 业务示例，如"管理员 / 普通用户，账号启用或禁用"> |
| 租户、组织或权限范围 | <TODO: 业务示例；项目不涉及时可省略> |
| 菜单、按钮和数据权限 | <TODO: 业务示例> |
| 业务数据状态 | <TODO: 业务示例，如"目标对象处于何种状态"> |
| 数据源与文件状态 | 数据表启用 / 失效；上传文件格式、字段、数据类型、范围 |
| 审批与签名配置 | <TODO: 业务示例> |
| 网络状态 | 正常 / 弱网 / 断网（异常场景必填） |
| 数据量要求 | <TODO: 业务示例> |
- 前置条件应能被测试人员独立复现，不能写"数据正常"。
- 前置条件只写执行用例前已经具备的环境、权限、数据和初始业务状态，不写后续操作或修复后的结果。
- 属于系统既有数据的前置条件，统一写成"系统已存在..."，不得写成"已准备基础数据"、"已准备对象"、"已准备用户"等容易混淆为测试步骤的表述。
- 前置条件不写系统默认能力或被测功能本身应提供的基础存在性，例如"系统存在导入模板"、"页面存在导出按钮"；这类内容应放在操作步骤或预期结果中验证。
- 前置条件不写以下冗余内容：

| 冗余内容 | 举例 | 原因 |
|---|---|---|
| 系统默认运行状态 | "网络正常"、"页面加载完成" | 除非用例验证异常或弱网场景，否则默认成立 |
| 系统间基础关联关系 | "<模块 A> 与 <模块 B> 已建立关联" | 属于系统运行前提，不是测试数据 |
| 业务对象完整上游链路 | "<对象>所属上游链路均已存在并启用" | 上游存在是下游存在的前提，不逐级列举 |
| 数据/文件符合要求断言 | "待上传文件符合配置要求" | 属于步骤中的操作或校验，不是前置条件 |
| 被测功能基础可用性 | "支持直接进入目标详情页" | 应在步骤或预期结果中验证 |
- 前置条件应简洁，不展开 PRD 中已有的字段格式、长度、枚举、唯一性等详细校验规则；需要说明文件或数据合法时，可写成"且格式符合要求"。字段规则的具体校验放到步骤或预期结果中验证，或拆成独立异常用例。
- 前置条件包含多个外部依赖时，必须使用有序编号（`1.`、`2.`、`3.`）逐条拆分，并在 Markdown 表格中用 `<br>` 换行展示；**单条建议不超过 50 字，超过则按外部依赖类型拆分为多个编号**；只有一个前置条件且不超过 50 字时可使用单句描述。
- 生成或评审用例时必须检查 `前置条件`、`用例步骤`、`预期结果` 是否重复表达同一件事：前置条件已写明的初始状态，步骤中不再用独立步骤复述；步骤中执行的动作，预期结果中不再重复描述操作过程。

**示例：**

| 写法 | 评级 |
|---|---|
| <TODO: 合格前置条件示例，包含用户、权限范围、业务数据状态> | 合格 |
| 数据正常 | 不合格 |
| 用户已登录 | 不完整（缺角色、权限范围和业务数据状态） |

### 创建人

- 必须填写创建人姓名或团队约定账号。
- 新生成用例固定填写 `AI Agent`，不随用户、模块或输入资料变化。

### 用例描述

- 用一句短语描述用例属性，推荐值：`正例`、`反例`、`边界`、`权限`、`UI`、`兼容`、`回归`、`联动`。
- 来自 UI 设计图、截图、原型图或界面交互说明的页面区域整体展示校验，优先使用 `UI`；`兼容` 仅用于浏览器、分辨率、弱网、多页签等环境适配或稳定性场景。
- 该字段用于快速筛选，不替代用例名称和预期结果。

### 用例步骤

- 使用有序编号（`1.`、`2.`、`3.`），每步一行，步骤间用换行分隔。
- **同一弹窗、同一页面或同一模块内的连续动作默认合并为一步，用逗号隔开继续写在一行**；合并后单步建议不超过 **50 字**，超过则按业务阶段拆分为多个编号步骤；跨页面、跨模块或不同业务阶段的操作必须拆分为独立步骤。
- 步骤数量不限，允许只有 1 步（如单纯打开页面查看展示、点击按钮触发生成等单动作场景）；步骤数较多时需检查能否按"同模块合并"原则压缩。

**示例：**

| 写法 | 评级 |
|---|---|
| <TODO: 不合格示例——同菜单+同弹窗拆成多步> | 不合格 |
| <TODO: 合格示例——菜单导航 1 步，弹窗内操作 1 步> | 合格 |
- 步骤只写用户动作、系统触发动作或必要的页面确认，不写纯场景陈述。
- 步骤不得写 `对照 PRD 检查`、`按需求核对`、`检查是否符合规则` 等不可直接执行的抽象动作；应改为打开具体页面、查看具体字段、下载具体文件或触发具体校验。
- 用例步骤应精炼，不把系统自动处理或被动等待单独写成步骤；例如上传或提交后，直接写查看结果。只有进度展示、异步任务、弱网、超时、重复提交等需要验证过程状态的场景，才保留等待、观察或持续触发类步骤。
- 步骤应包含操作对象和动作，例如：
  - 合格：`1. 登录目标系统`
  - 合格：`2. 进入目标列表，点击某对象的"操作"按钮`
  - 合格：`3. 上传符合要求的文件并点击"提交"`
  - 不合格：`1. 处理数据`（动作与对象不明确）

### 预期结果

- 必须包含页面表现、数据状态或业务状态中的至少一项，不能少于 10 个字。
- 不允许使用以下空泛表达（单独作为唯一预期时）：

| 禁止写法 | 原因 |
|---|---|
| 页面展示正常 | 无可验证标准 |
| 功能可用 | 无法操作验证 |
| 结果正确 | 无法判断正确标准 |
| 按需求展示 | 依赖人工理解 |
| 展示成功 / 操作成功 / 通过 | 过于笼统 |

- 推荐写法：明确说明页面元素、数量、跳转目标、提示文案或数据变化。
- **同一弹窗、同一页面或同一模块内的校验点默认用逗号或分号串联成一句**；串联后建议不超过 **50 字**，超过则按验证目标拆分为多个编号；跨页面区域、不同业务状态、数据写入与权限拦截等真正独立的结果，必须拆分为独立编号。

**示例：**

| 写法 | 评级 |
|---|---|
| <TODO: 不合格示例——同一弹窗拆成多行> | 不合格 |
| <TODO: 合格示例——同一弹窗内串联为一句> | 合格 |
- 避免使用 `按...规则`、`根据...规则`、`符合...规则` 作为预期结果表达，应直接写最终可观察结果。
- 预期结果涉及数据变化或页面更新时，必须写明变化后的具体内容，不得只写"对应更新""相应变化"等笼统表达。
- 涉及元素展示的用例，预期结果必须对照 PRD / UI 原文列表逐条核对，列出所有页面元素，不得遗漏；遗漏任何一项视为用例不完整。
- 涉及核心业务链路时，推荐同时说明上下游状态变化、引用关系、操作日志或签名记录。
- 仅验证页面元素、按钮、字段、布局、说明文案或静态展示时，按区域生成 `UI校验` 用例，`用例描述` 填写 `UI`，预期结果列出关键元素，不写"页面展示正确"。
- 同一页面区域只保留一条整体 `UI校验` 用例；字段、按钮、文案、布局、默认状态合并其中，除非该元素有独立交互、业务规则、异常提示或边界限制。

**推荐示例：**

| 场景 | 推荐写法 |
|---|---|
| 正向登录系统 | 登录成功，页面跳转至首页，顶部展示当前用户信息和可访问范围 |
| <TODO: 业务正向示例> | <TODO: 业务示例预期结果> |
| <TODO: 业务异常示例> | <TODO: 业务示例预期结果> |
| <TODO: 业务边界示例> | <TODO: 业务示例预期结果> |

### 备注、标签和关联字段

- 新生成或追加的用例，`备注` 必须填写来源，不允许为空或填写 `无`。按来源类型选择写法：

| 来源类型 | 写法 | 示例 |
|---|---|---|
| 需求文档、PRD、验收标准或用户补充需求 | `来源：prd` | `来源：prd` |
| UI 设计图、截图、原型图或界面交互说明 | `来源：UI图` | `来源：UI图` |
| 参考用例模板 | `来源：xxx.md`（必须写清具体模板文件名） | `来源：<模板文件名>.md` |
