# 覆盖维度规则

## 使用原则

生成或补充测试用例时，先按本文件评估覆盖范围，再结合 PRD、UI 设计图、知识库和参考用例生成具体场景。

- 每个功能模块必须评估"必须覆盖维度"，只保留与当前需求有直接关系的场景。
- 核心业务链路重点覆盖正向主流程、异常拦截、状态流转、数据一致性、关键操作追溯和跨模块联动。
- 涉及专项能力时，按"专项规则路由"读取对应规则文件，不在本文件展开业务细节。
- 本文件只回答"要从哪些维度评估覆盖"，不承担优先级、难度或具体模块用例模板职责。

---

## PRD 章节归属原则

**PRD 章节归属原则**（强制）：PRD 当前章节内写明的所有需求点（含表格、文案模板、字段定义、状态流转、跨模块联动等）必须在当前章节对应的用例文件中生成覆盖。即使需求点的触发条件跨模块，只要 PRD 在当前章节描述，用例就归属当前章节，不得推到其他业务模块。

---

## 适用性判定

**通用覆盖维度不是"必生成清单"**。每个维度必须先做适用性判定，PRD 未明确的维度不生成用例：

- **PRD 明确涉及** → 生成对应测试用例，并在覆盖率对照表列出覆盖关系
- **PRD 未明确** → 不生成（即使维度本身在系统中存在）；在覆盖率对照表写 `不适用：PRD 未明确 XXX 规则`
- **PRD 暗示但未明确** → 写入需求问题清单或覆盖率对照表回查项

**禁止**：跳过适用性判定、按"维度存在就要覆盖"或"通用维度必生成"生成用例；禁止从覆盖维度反推 PRD 未写的具体规则作为预期结果。

通用覆盖维度用于生成前评估，不代表每个需求都必须生成所有维度的用例。

| 判定结果 | 条件 | 处理方式 |
|---|---|---|
| 适用 | PRD、UI 图、知识库或参考模板明确涉及该能力 | 生成对应测试用例，并在覆盖率对照表列出覆盖关系 |
| 不适用 | 当前需求不涉及该能力，且知识库未声明该项目具备该能力 | 不生成用例；如属于高风险维度，在覆盖率对照表写 `不适用：<原因>` |
| 待确认 | PRD 暗示该能力但未说明规则、范围或预期结果 | 直接影响执行结论时写入需求问题清单；仅影响覆盖范围时写入覆盖率对照表回查项 |
| 资料缺失 | UI 图、参考模板或知识库缺失，但 PRD 已明确需求 | 按 PRD 生成，缺失资料写入元信息的生成假设 |

以下能力只在项目实际存在时生成，不得为了追求覆盖率强行扩展：

- 审批、签名、发布、生效、版本、归档等流程治理能力。
- 租户、组织、产品线或数据域隔离。
- 操作日志、审计追踪、消息通知、待办、异步任务。
- 导入、导出、模板下载、错误报告。
- 第三方系统同步、外部接口、定时任务和批处理。

## 必须覆盖维度

| 维度 | 覆盖重点 | 对应用例类型 |
|---|---|---|
| 正向主流程 | 用户按角色权限完成标准业务操作，如创建、提交、审批、生效、发布等 | 正例 |
| 异常流程 | 接口失败、网络中断、数据源失效、依赖不可用、审批失败、异步任务失败等异常处理 | 反例 |
| 边界值 | 字段长度、文件大小、数量、周期范围、阶段数、阈值、区间等临界值 | 边界 |
| 字段与数据校验 | 必填项、格式、唯一性、数据类型、单位、范围、完整性、字段匹配和无效引用校验 | 反例 / 边界 |
| UI 设计图覆盖 | 页面区域、标题说明、字段、默认值、禁用态、按钮、表格列、结果区、提示文案和布局与 UI 设计图一致 | UI |
| 权限与数据隔离 | 未登录、登录态过期、无菜单权限、无按钮权限、无数据权限、越权访问、跨租户 / 跨组织 / 跨数据域访问拦截 | 权限 |
| 状态流转 | 草稿、待审批、审批中、已批准、已退回、已关闭、已生效、已复核、已完成、已放行、已归档等 QRS 业务状态变化和可操作范围 | 正例 / 反例 |
| 数据一致性 | 上下游对象之间的引用、状态和版本一致 | 联动 / 回归 |
| 关键操作追溯 | 操作日志、签名记录、审批意见、版本历史、操作前后值、失败原因和重试记录 | 回归 / 权限 |
| 跨模块联动 | 当前模块操作对下游模块、待办和通知的影响 | 联动 |
| 兼容性与稳定性 | 不同浏览器、分辨率、弱网、重复点击、页面刷新、返回、多页签、长耗时任务恢复 | 兼容 / 回归 |

## 字段定义表与状态对照表覆盖规则

"必须覆盖维度"给出通用评估口径，本节进一步细化字段定义表和状态对照表两类 PRD 表格的行级覆盖要求。生成用例前必须按本节展开 checklist，避免只覆盖表格"某一行"而漏掉该行内部的多个校验规则或多个操作按钮。

### A. 字段定义表逐字段覆盖

生成用例前，必须按字段定义表逐字段列出校验规则清单，每条规则对应至少 1 条用例：

| 字段规则类型 | 覆盖要求 |
|---|---|
| 必填字段 | 正例（选了能提交成功）+ 反例（不选 / 清空时拦截）+ 下拉选项 UI 校验（如有下拉） |
| 格式/长度规则（如 2-100 字符） | 边界用例（最小值成功、最大值成功、超出失败） |
| 范围规则（如 ≥当前日期、≤计划结束日期） | 边界用例（临界值成功、超出失败） |
| 唯一性规则 | 唯一值成功 + 重复值失败 |
| 引用存在性规则（如只能选已生效计划） | 合法引用成功 + 非法引用不可选或失败 |
| 条件必填字段 | 条件满足时必填校验 + 条件不满足时非必填校验 |
| 非必填字段 | 留空可提交成功 + 全填可提交成功 |

关键约束：

- **不得只写正例**：每个必填字段必须配套反例，不能只有"选了能成功"的正例。
- **不得用一条用例概括所有字段**：每个字段的每个规则独立覆盖，除非验证目标完全一致。
- **"编辑同新增"对齐**：PRD 写"编辑同新增"时，编辑入口必须复制新建的完整校验体系，不得遗漏。

### B. 状态×操作矩阵覆盖

生成用例前，必须将状态与操作按钮对照表展开为二维矩阵，每个"状态×可用操作"格子都要有对应的"操作成功"正例：

| 矩阵维度 | 覆盖要求 |
|---|---|
| 状态×操作按钮 | 状态对照表中每个状态的每个可用操作按钮，都要有"点击该按钮后操作成功"的正例用例 |
| 按钮范围校验 | 保留现有"可执行操作按钮范围"回归用例（验证按钮存在/不存在），但这不能替代操作成功正例 |
| 跨状态一致操作 | 查看、去执行等跨状态行为一致的操作，可在一条用例中覆盖多个状态（步骤里分别验证），不必每状态一条 |

关键约束：

- **按钮范围用例 ≠ 操作成功用例**："操作列含编辑按钮"只验证按钮存在，不验证点击编辑后能保存成功。
- **关闭操作要覆盖所有可关闭状态**：不能只写一个状态的关闭，每个可关闭状态都要有独立正例或合并覆盖。

## 模块专项覆盖来源

本文件只定义通用覆盖维度。模块业务细节以知识库和参考用例为准。

| 模块类型 | 优先读取 |
|---|---|
| 核心业务主链路 | `knowledge_base/core_flows/` 下对应流程文件 |
| 登录、菜单权限、按钮权限、数据权限、租户 / 组织 / 数据域隔离 | `knowledge_base/user_roles.md`、`knowledge_base/core_flows/` 下登录/权限相关流程 |
| 具体菜单模块 | `testcase_templates/modules/menu_index.md` 命中的参考用例 |

## 项目高风险场景（按需补充）

QRS 系统以下场景一旦涉及，必须评估并生成对应用例；具体优先级以 `priority_rules.md` 为唯一判定来源。

| 高风险场景 | 覆盖重点 |
|---|---|
| 未生效、已失效或已锁定的回顾对象 / 模版被继续引用或编辑 | 状态拦截、数据不被写入、提示可验证（DF-001 / DF-004 / DF-007 引用链路） |
| 跨部门 / 跨数据域、无菜单 / 按钮 / 数据权限或接口越权（QA 专员、工艺工程师、生产人员、QC 等角色越权） | 入口隐藏 / 禁用、接口拦截、业务数据不泄露 |
| 关键动作（审批、复核确认、放行、归档、退回）缺少电子签名或操作日志 | 操作人、时间、原因、前后值、版本历史（ALCOA+ 原则） |
| 上游数据（已复核数据、分析结果）失效后下游（分析、报告）仍生成有效结果 | 阻断、原因展示、状态不误更新（DF-003 / DF-005 / DF-006） |
| 批量导入 / 批量复核 / 批量审批重复提交 | 不产生重复数据、重复任务或重复记录 |
| 规格限（USL / LSL）配置错误或缺失导致 Cpk / Ppk 计算异常 | 阻断、提示可验证、不产出错误分析结果 |
| 审计追踪日志缺失或被篡改 | 操作可追溯、日志不可篡改、满足 GMP 合规要求 |

## 专项规则路由

命中以下场景时，必须读取对应专项规则，再生成或补充用例。未命中专项规则路由时，不得读取对应专项规则文件；不得为了泛化覆盖、参考风格或扩大覆盖面主动读取无关专项规则。

| 场景 | 必读规则 |
|---|---|
| 涉及 Excel / CSV / 模板文件导入、批量导入、导入模板、错误报告 | `generation_rules/import_coverage_rules.md` |

> QRS 当前仅有导入专项规则。如后续新增其他专项规则（如审批流、电子签名、ALCOA+ 数据完整性专项等），应在此表中同步登记路由关系。

专项规则只补充覆盖场景，不替代：

- 字段写法遵守 `testcase_writing_guidelines.md`。
- 优先级判定遵守 `priority_rules.md`。
- 难度判定遵守 `difficulty_level_rules.md`。
