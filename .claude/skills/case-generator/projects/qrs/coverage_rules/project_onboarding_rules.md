# 项目接入规则

本文件定义通用测试用例生成器接入新业务系统时的最小配置、资料优先级和缺失信息处理方式。生成具体需求用例前，若发现项目知识库仍为空或关键业务上下文缺失，应按本文件判断是否继续生成、写入生成假设或列入需求问题清单。

## 最小接入信息

新项目至少应补充以下信息；缺失时不阻塞读取 PRD，但会影响用例质量：

| 信息项 | 建议位置 | 用途 | 缺失处理 |
|---|---|---|---|
| 系统名称与业务目标 | `knowledge_base/project_overview.md` | 判断模块边界和业务价值 | 写入元信息生成假设 |
| 用户角色与权限模型 | `knowledge_base/user_roles.md` | 生成权限、菜单、按钮、数据隔离用例 | 涉及权限预期时列入需求问题清单 |
| 核心业务模块列表 | `knowledge_base/project_overview.md` | 确定输出文件名和分组 | 按 PRD 原文归类，并写生成假设 |
| 核心业务流程 | `knowledge_base/core_flows/<flow_name>.md` | 补充跨模块联动、状态流转和回归场景 | 仅按 PRD 明确范围生成 |
| 业务术语与状态枚举 | `knowledge_base/business_glossary.md` | 保证字段、状态和预期用词准确 | 不自行推断枚举值 |
| 项目追踪字段 | `scripts/business_constants.py` | 扩展用例表尾部字段 | 未配置时使用默认通用表头 |

## 资料优先级

生成用例时，事实来源按以下顺序判定：

1. 用户本次明确指定的需求章节、补充说明或验收标准。
2. 同章节或用户指定章节下的 UI 图、截图、原型和交互说明。
3. PRD 中被当前章节引用的其他章节。
4. `knowledge_base/` 中的项目背景、角色、术语和核心流程。
5. `testcase_templates/` 中的参考模板和通用模板。
6. 通用规则文件中的覆盖维度、优先级和难度判定。

参考模板只提供写法、覆盖思路和去重依据，不得覆盖 PRD 明确要求。

## 缺失信息处理

| 缺失类型 | 处理方式 |
|---|---|
| 不影响具体预期的背景信息缺失 | 继续生成，在元信息 `生成假设` 中说明 |
| 权限、数据隔离、状态流转或字段规则缺失 | 生成待确认用例，并写入需求问题清单 |
| 专项覆盖项是否适用不明确 | 写入需求覆盖率对照表的 `[回查]` 记录 |
| UI 图缺失 | 不写入需求问题清单，在元信息和覆盖率回查中说明 |
| 参考模板缺失 | 不阻塞生成，优先使用通用覆盖维度和通用模板 |
| PRD 与 UI 图冲突 | 写入需求问题清单；影响核心预期时先询问用户 |

## 接入完成检查

新项目正式生成前，建议运行：

```bash
python scripts/validate_project_config.py
```

若仍有真实业务样例、旧项目词、Word 临时锁文件或输出产物污染，应先清理或归档，再生成正式用例。
