# 审计追踪专项覆盖规则

## 使用原则

涉及审计追踪、电子签名记录、操作日志、操作前后值、详情导出或合规追溯时，必须先读取本文件，再生成或补充用例。

- 本文件只定义审计追踪专项覆盖场景；优先级仍按 `priority_rules.md` 判定，难度仍按 `difficulty_level_rules.md` 判定。
- 只生成与当前模块、当前操作和当前审计记录直接相关的场景，不因为涉及审计追踪而扩展无关模块、无关操作或无关字段。
- 审计追踪用例必须明确触发来源：业务模块、用户权限、操作动作，以及该操作后产生或不产生审计记录。
- 审计追踪权限统一按审计追踪菜单权限理解，不单独拆分审计追踪查看权限、导出权限或按钮权限。
- 审计追踪字段、操作类型、详情内容、筛选条件和导出格式必须以当前 PRD / UI / 参考用例为准；资料未说明时，不得自行补充确定结论，应在需求覆盖率对照表中标记需要确认。
- PRD 明确要求的审计追踪核心记录和详情字段准确性可按合规发布阻断项判为 `P0`；权限访问控制、详情导出、搜索筛选、操作类型下拉和普通展示体验默认不因涉及审计追踪自动升为 `P0`。

## 默认覆盖项

| 覆盖项 | 检查要求 |
|---|---|
| 列表记录 | 触发当前模块支持审计的操作后，审计追踪列表新增对应记录；记录应出现在列表第一行，至少校验操作人、操作时间、所属模块、操作类型和受影响对象 |
| 操作时间 | 优先按模板口径校验为 `yyyy-mm-dd hh:mm:ss`；若当前 PRD / UI 明确其他格式，以当前资料为准 |
| 受影响对象 | 使用实际业务对象名称；批量操作类受影响对象命名以当前 PRD / UI / 参考用例为准 |
| 详情入口 | 点击受影响对象进入详情；详情应包含 `基本信息` 和 `内容详情` 两类信息，弹窗或详情页形式以当前 UI 为准 |
| 详情信息 | 基本信息和内容详情字段必须按当前模块 PRD / UI 确定，不同模块不得套用固定字段清单；预期结果不得只写“显示操作模块、操作类型”等字段名，当前模块和操作类型已知时必须写明具体值 |
| 前后值对比 | 编辑、配置、状态变更等操作需校验修改前值和修改后值；未变化字段不应被记录为变更 |
| 批量操作 | 批量删除、批量导入、批量导出等场景需校验数量、每条明细和受影响对象命名规则 |
| 无变更操作 | 进入编辑、配置或详情页面但未做任何实际修改时，不应新增审计追踪记录 |
| 详情导出 | 默认按 PDF 校验，文件名优先为 `<受影响对象>.pdf`，导出内容需与页面详情展示一致；当前 PRD / UI 指定其他格式或文件名时，以当前资料为准 |
| 搜索筛选 | 至少评估模块筛选、操作类型筛选、受影响对象关键字筛选和组合搜索；组合搜索需验证搜索条件保持已选值，列表仅展示同时满足条件的审计记录 |
| 操作类型下拉 | 选项需随所选模块变化，只展示当前模块实际支持的操作类型，不得混入其他模块操作类型 |
| 权限控制 | 仅按审计追踪菜单权限控制访问；无审计追踪菜单权限时，入口、列表、详情、导出按钮、接口和数据访问均应受限，不泄露审计数据；不单独生成无审计追踪导出权限场景 |
| 站点隔离 | 公共管理站点和业务站点审计记录按所属站点隔离，跨站点用户不得查看或导出无权访问的数据 |

## 分组与命名

- 审计追踪专项用例的 `一级分组` 优先填写 `审计追踪`。
- `二级分组` 填写被审计的业务模块。
- `三级分组` 填写当前模块实际操作类型、子流程或查询场景。
- 如审计详情、筛选条件、详情导出等还需继续细分，可在 `三级分组` 后增加 `四级分组`、`五级分组` 等，分组值填写对应子功能名称。
- 用例名称应写清审计验证目标，优先使用 `<操作>成功，审计追踪列表新增对应记录`、`查看<操作>审计追踪详情，验证数据正确`、`导出<操作>审计追踪详情，验证导出内容一致`。
- 相同业务模块下，同一操作类型的列表记录、详情、导出和不产生日志场景应相邻放置。

## 生成前必读 checklist

本节把散落在前述章节的"必须覆盖项"和"易漏反例项"集中成一张表，**生成用例前和生成后回查时必须逐条勾对**。当前 PRD / UI 明确不适用的项可在需求覆盖率对照表写明不适用原因后跳过，但不得静默遗漏。

### 必须覆盖项（按被审计模块的完整链路）

| # | 必须覆盖项 | 至少条数 | 易漏点 |
|---|---|---|---|
| 1 | 列表记录新增（操作人、操作时间、所属模块、操作类型、受影响对象） | 每个被审计操作 1 条 | 字段值必须写具体，不得只写"显示操作模块、操作类型"等字段名 |
| 2 | 详情入口与详情信息（基本信息 + 内容详情） | 每个被审计模块 1 条 | 字段必须按当前模块 PRD/UI 写明具体值，不同模块不得套用固定字段清单 |
| 3 | 详情导出（默认 PDF，文件名 `<受影响对象>.pdf`） | 每个被审计操作 1 条 | 导出内容必须与页面详情展示一致；PRD/UI 指定其他格式时以资料为准 |
| 4 | 搜索筛选（模块筛选、操作类型筛选、受影响对象关键字筛选、组合搜索） | 至少 1 条覆盖 4 种 | 组合搜索必须验证搜索条件保持已选值，列表仅展示同时满足条件的记录 |
| 5 | 操作类型下拉随所选模块变化 | 1 条 | 只展示当前模块实际支持的操作类型，不得混入其他模块操作类型 |
| 6 | 无审计追踪菜单权限，入口/列表/详情/导出按钮/接口/数据全链路受限 | 1 条 | 不单独生成无审计追踪导出权限场景；权限统一按审计追踪菜单权限 |
| 7 | 跨站点审计记录按所属站点隔离 | 1 条 | 跨站点用户不得查看或导出无权访问的审计数据 |

### 条件性必须项（视操作类型生成）

| # | 触发条件 | 必须覆盖项 | 易漏点 |
|---|---|---|---|
| 8 | 编辑、配置、状态变更等操作 | 修改前值和修改后值对比 | 未变化字段不应被记录为变更 |
| 9 | 批量删除 / 批量导入 / 批量导出 | 数量、每条明细、受影响对象命名规则 | 不得只写"批量操作审计记录正确" |

### 易漏反例项（必须生成或在预期中校验）

| # | 反例项 | 说明 |
|---|---|---|
| 10 | 无变更操作不应新增审计追踪记录 | 进入编辑、配置或详情页面但未做任何实际修改时，不应新增审计追踪记录 |
| 11 | 不扩展无关模块、操作或字段 | 只生成与当前模块、当前操作和当前审计记录直接相关的场景 |

**回查要求**：

- 生成用例后，必须对照本表逐条勾对；命中但未生成的项必须补充用例。
- 确认不适用的项必须在需求覆盖率对照表中写明不适用原因，不得静默遗漏。
- 本表是"必须覆盖项"的最小集，PRD / UI 额外明确的审计场景仍按业务需要生成，不受本表限制。
