# 数据分析专项覆盖规则

## 使用原则

新增或补充分析方法、数据分析详情页、分析结果、替换数据源或一键分析用例时，必须先读取本文件。

- 除 PRD 显式需求外，还需评估本文件中的默认质量风险。
- 当前 PRD / UI 明确不适用的默认覆盖项可以不生成，但需在需求覆盖率对照表中记录不适用原因；适用的核心链路、异常原因和结果正确性场景不得遗漏。
- 本文件只定义数据分析专项覆盖场景；优先级仍按 `priority_rules.md` 判定，难度仍按 `difficulty_level_rules.md` 判定。

---

## 通用能力

- 读取 `knowledge_base/core_flows/data_analysis_flow.md`，确认三类入口、分析方法范围和结果处理差异。
- 按当前入口和分析方法覆盖文件校验、插件状态、算法配置、字段配置、动态表单和开始分析前校验。
- 覆盖数据分析工作台、监控项目、相关性分析汇总三类入口的分析方法范围差异。
- 覆盖当前适用入口的结果处理：数据分析工作台自动保存分析结果；监控项目和相关性分析汇总推送至业务系统。推送用例校验推送状态、分析状态、分析结果名称，预览分析结果详情；预览分析结果时只验证 `详细结论` 和 `输入结论` 显示正确。
- **监控项目 / 相关性分析汇总列表中分析结果名称清空**：监控项目或相关性分析汇总列表中，项目分析状态为"未分析"时，列表中"分析结果名称"字段清空。数据分析详情页分析图生成区域中的分析结果名称不清空，仍保留历史名称。
- **数据分析详情页通用能力归"数据分析工作台"下**：配置项、分析结果展示、详情页内异常处理、数据源预览、PNG/JPG 导出等详情页内通用能力，统一归入 `数据分析工作台 / 数据分析 / ...` 下；监控项目和相关性分析汇总下只写入口特有用例（新增分析方法配置、推送、状态更新、报告插入、一键分析、变量带出、置灰限制等）。
- 生成分析方法所属分类用例时，分类名必须取自 PRD 中当前分析方法所在的 `选择算法-<分类名>` 上级标题里的 `<分类名>` 原文，不得自行概括或改写。
- 变量名称、变量数量、必填性、结果区域字段必须以当前 PRD 或 UI 设计图为准，不得沿用其他分析方法或参考模板中的字段名。PRD 未明确时必须查看 UI 图，PRD 和 UI 图均未明确或部分明确但未明确具体变量角色时，必须标记【待确认】并写入需求问题清单，不得从 Minitab 或通用知识推断为确定结论。
- 生成结果页用例前，按当前 PRD / UI 实际存在的页面区域区分 `分析流程`、`分析方法`、`分析图表`、`过程数据`、`判异检查`、`参考线检查`、`输入结论`，不得跨区域复用字段或强行补不存在的区域。PRD 中的"算法配置"统一写为"分析方法"。
- 分析结果区域的说明文字必须按 PRD 原文写具体内容，不得笼统写"分析结果说明"；PRD 原文有明显笔误时可修正。
- 涉及结果展示的用例（如不同数据类型、更改配置后分析、一键分析成功等），预期结果必须对照 PRD / UI 列出所有结果区域和变量角色，不得遗漏；遗漏某个结果区域或变量角色视为用例不完整。
- 算法结果、过程数据、控制限等计算一致性用例，默认按第三方软件（如 Minitab）或测试团队既有对比口径生成；PRD 未展开算法公式本身不作为需求问题写入需求问题清单。只有用户明确说明没有第三方对比基准或无法提供业务确认值时，才将预期结果无法判定写入需求问题清单。

### 参考模板使用

- 参考已有分析方法模板时，只借鉴分组组织和前置条件、步骤、预期结果的表达风格；当前方法不具备的配置、结果区域、异常或联动能力不得迁移。
- 分组列按当前用例的自然业务层级填写，不为对齐参考模板强行补齐固定层级；没有对应业务节点时留空。
- 分组名称、入口、权限、变量角色、提示文案和结果区域以当前 PRD / UI / 业务流程为准；参考模板内容仅在业务含义一致时复用。
- 步骤和预期结果不得使用来源不清、无法在页面定位的泛化表述；引用 PRD 上级标题、通用流程或专项规则时，应在备注或覆盖率对照表中保留真实来源。
- **配置项三级分组下按具体配置项细分四级分组**：当"数据分析工作台 / 数据分析 / 配置项"下涉及多个独立配置项（≥6 条用例）时，必须按 PRD 配置项表的"配置项"列原文细分四级分组（如"放入【定量】变量"、"子组大小"、"参考线设置"、"判异检查"），便于定位与回归。任一四级分组至少 2 条用例；只有 1 条用例的配置项按"X 与 Y"形式并入相邻同级分组（如"估算标准差"和"旋转角度"合并为"估算标准差与旋转角度"）。其他三级分组（分析结果、异常流程等）一般不细分四级。

## 分析方法默认覆盖项

新增或补充任一分析方法时，默认检查以下项；若当前 PRD / UI 明确不适用，可以不生成，并在需求覆盖率对照表中写明不适用原因。本节覆盖手动开始分析和结果查看链路；下方“一键分析”章节覆盖数据替换或数据处理后的重新分析链路。

| 默认覆盖项 | 检查要求 |
|---|---|
| 变量区域基础交互 | 按当前分析方法支持的变量角色、变量类型和变量数量限制生成；不支持定类变量时需校验拦截，支持定类变量时不得生成"定类变量不支持"类用例。**变量数量边界必须覆盖**：当某变量角色接收多个变量（如"接收多个定量变量"）时，除验证"可添加多个"外，必须额外生成一条边界用例验证"只添加1个变量时分析成功"，确认变量数量下限可用；当某变量角色只接收1个变量时，必须验证"已有变量时不允许添加第二个"。 |
| 未分析项字段匹配（变量带出） | 必须生成两条用例：字段匹配上→自动带出原字段；字段匹配不上→不带出、配置为空。**变量带出只针对主分析变量，非主分析变量（如连续预测变量、分组变量等辅助维度）不参与带出**，带出用例的预期结果只校验主分析变量的带出情况。参考 `monitoring_items_i_chart_template.md`。 |
| 数据源预览 | 按当前入口进入数据分析详情页后，数据源预览入口、预览内容、字段和数据行显示正确，不固定为监控项目入口 |
| PNG/JPG 导出 | 当前分析方法支持导出图片时，导出内容应仅包含分析图表，不覆盖分析流程、详细结论、过程数据等结果页其他区域 |
| 分析结果标题 | PRD / UI 未明确命名规则时，默认按 `<主分析变量>_<分析方法名称>` 生成作为假设，必须标记【待确认】并写入需求问题清单。主分析变量按当前方法实际变量角色判断，辅助维度不作为主分析变量；多个主分析变量时用顿号连接后再拼接方法名称。 |
| 小数位数默认选项 | 当前分析方法存在小数位数配置时，需校验默认值和下拉选项；**具体默认值和选项范围必须以当前分析方法的 PRD 为准，PRD 与 UI 图冲突时以 PRD 为准** |
| 小数位数影响范围 | 小数位数对各字段小数位的影响范围必须以 PRD 为准，逐字段判断是全部固定、部分受影响还是全部受影响。不得从参考模板或其他分析方法推断；PRD 未明确时写入需求问题清单。小数位数影响范围集中在专门用例中验证，不得散落到其他用例的预期结果里。结果区域过多时按区域拆分为多条独立用例。 |
| 不同数据类型 | 必须优先以当前分析方法的 PRD / UI、算法限制和字段类型为准。**小数列、负数列、混合型数据列默认生成正例**：PRD 未禁则必生成，验证分析成功且计算结果正确；其中**混合型数据列**的测试数据需包含当前方法支持的所有数据类型（整数、小数、负数、% 等）；PRD 明确不支持则生成反例拦截用例。**百分位数列、时:分:秒列**作为风险覆盖维度评估，资料未明确限制时不得直接断言全部支持。 |
| 更改配置后分析 | PRD 明确"已生成分析结果可在原有配置基础上修改或继续分析"（分析流程限制）时必须覆盖。最少覆盖两类用例：1) 点击"更改配置"按钮，原配置项（变量、参数、小数位数等）数据正确回显；2) 修改变量或参数后重新分析，生成最新结果。归入 `数据分析工作台 / 数据分析 / 更改配置` 三级分组。参考 `monitoring_items_box_plot_template.md`、`monitoring_items_paired_t_template.md`。 |

## 一键分析

新增或补充支持数据替换或数据处理，并会对当前已生成分析结果重新分析的分析方法用例时，一键分析为默认覆盖项，不以 PRD 是否显式描述为前提。若当前 PRD / UI 明确没有一键分析入口或不支持重新分析，应在需求覆盖率对照表中记录不适用原因，不强行生成一键分析用例。

### 生成前提

- 先读取 `knowledge_base/core_flows/one_click_analysis_flow.md`。
- 参考以下模板中的一键分析相关用例风格：
  - `testcase_templates/modules/business_site/scheme_execution/monitoring_items_i_chart_template.md`
  - `testcase_templates/modules/business_site/scheme_execution/monitoring_items_box_plot_template.md`
  - `testcase_templates/modules/business_site/scheme_execution/monitoring_items_paired_t_template.md`
- 一键分析默认先覆盖触发方式：`数据替换`、`数据处理`。
- 已存在未分析项时，还需覆盖未分析项在数据替换或数据处理后重新参与一键分析的场景。
- 一键分析失败原因按当前分析方法的 PRD / UI、变量类型和算法限制筛选，不固定套用下方所有“候选失败原因”类别。
- 生成修复成功类用例前，必须先判断失效字段是否仍保留在配置中；字段已被清空的场景不得生成“一键分析修复成功”用例。
- 修复成功类用例名称必须明确包含 `一键分析成功`，不得只写“修复后成功”。

### 分组和回查口径

- “默认覆盖项”和“候选失败原因”是生成后回查清单，回查结果写入需求覆盖率对照表。
- 分组列是测试用例表中的组织字段，不得把“适用失败原因”等回查项直接当作用例分组名。
- 一键分析用例通常使用二级分组 `一键分析`；三级分组及后续分组按实际场景填写，如 `数据替换`、`数据处理`、`一键分析结果提示`、`未分析的图参与一键分析`、`定类变量暂不支持分析`、`字段删减`、`其他原因`，必要时可用 `四级分组` 承载更细子功能。
- **一键分析仅在 `监控项目` / `相关性分析汇总` 下覆盖**：归入对应一级分组的 `一键分析` 二级分组（如 `监控项目 / 一键分析 / ...`）。`数据分析工作台` 下不生成一键分析用例。
- **变量带出仅在 `监控项目` / `相关性分析汇总` 下覆盖**：归入对应一级分组的 `变量带出` 二级分组。`数据分析工作台` 下不生成变量带出用例。

### 候选失败原因

| 检查顺序 | 候选失败原因 | 适用条件 |
|---|---|---|
| 1 | 定类变量暂不支持分析 | 仅适用于当前分析方法不支持定类变量的场景；若当前分析方法支持定类变量，不得生成该类用例。 |
| 2 | 字段删减 | 当前分析方法依赖字段在替换或处理后的数据中不存在。 |
| 3 | 正态性变化 | 仅适用于过程能力分析（正态）和过程能力分析（非正态）。 |
| 4 | 阶段内容不一致 | 仅适用于存在阶段配置或参考线依赖阶段的分析方法。 |
| 5 | 其他原因 | 当前分析方法 PRD / UI 明确的样本量、数据量、变量值、阶段数等异常。 |

### 默认覆盖项

| 默认覆盖项 | 检查要求 |
|---|---|
| 触发方式 | 数据替换：必须覆盖至少两种数据来源（如系统集成表、数据处理表、本地上传表）；数据处理：必须覆盖筛选、转换、剔除等至少两种子场景。参考 `monitoring_items_i_chart_template.md`。 |
| 结果提示 | 覆盖全部成功、部分成功部分失败、全部失败提示，以及全部失败时确认清空和取消保留状态 |
| 未分析项参与一键分析 | 结合当前分析方法 PRD / UI 的异常 / 边界流程处理生成，覆盖字段仍可匹配、数据修复、样本量 / 数据量 / 变量值等条件恢复后重新参与一键分析，以及仍不满足条件时保持未分析并提示原因；字段保留与修复成功关系按下方“未分析项修复判定”处理 |
| 多变量配置项 | 当前分析方法存在多个变量配置项时，按 PRD 或 UI 设计图列出每个变量的缺失、字段不匹配、字段清空和字段保留后的数据修复场景；变量缺失类用例必须按变量拆分，不得只生成一条泛化用例 |
| 适用失败原因 | 从上方候选失败原因表中筛选当前分析方法适用的原因，覆盖未分析状态、悬停原因、配置保留 / 清空、修复后重新分析和仍失败时的处理 |
| 原因优先级 | 多个失败原因同时命中时，只展示最高优先级原因；需要按当前方法适用原因构造优先级校验，不强行套用不适用原因 |
| 手动分析与一键分析共用异常 | 同一异常若既适用于手动开始分析又适用于一键分析，需分别覆盖手动分析拦截和一键分析下的未分析状态，不得只生成其中一种入口 |

### 未分析项修复判定

| 失败状态 | 字段是否仍保留 | 是否可生成一键分析修复成功用例 | 生成口径 |
|---|---|---|---|
| 字段删减、字段缺失、字段匹配失败、匹配上定类变量导致变量或子组大小被清空 | 否 | 否 | 只生成未分析、字段清空、需手动配置类用例；不得生成“修复后一键分析成功” |
| 变量字段仍保留，但变量值不满足算法限制 | 是 | 是 | 可生成数据处理或数据替换修复后 `一键分析成功` 用例，例如 P控制图纵轴变量不包含整数修复后一键分析成功 |
| 样本量 / 数据量不足，但变量字段仍保留 | 是 | 是 | 可生成补足数据后 `一键分析成功` 用例 |
| 缺陷数与子组数量关系不合法，但相关字段仍保留 | 是 | 是 | 可生成修正缺陷数或子组数量后 `一键分析成功` 用例 |

上表"匹配上定类变量"涵盖所有变量类型变为不适用的等价措辞，包括：变为定类变量、定量变量变定类、变为非定量、图形变量变为非定量等。这类场景下变量字段会被清空，不属于字段保留类，不得生成一键分析修复成功用例。`validate_cases.py` 的字段已清空类识别同样覆盖这些等价措辞。

字段已被清空后重新手动选择变量或子组大小，属于手动更改配置或重新分析，不归入未分析项参与一键分析修复成功。

`validate_cases.py` 的 `one_click_missing_field_retained_repair` 规则会按上表通用强制配对：扫描所有一键分析「未分析」失败用例，排除字段已清空类（名称含字段删减、字段缺失、匹配上定类变量、变量缺失、已清空等关键词）和修复成功类自身后，对剩余的字段保留类失败用例动态提取原因核心，检查是否存在对应的「一键分析成功」修复用例；不匹配时报 WARN。该规则不依赖硬编码的具体失败原因，新增任何分析方法的字段保留类失败都适用。

### 未分析的图参与一键分析写法

本类用例可参考 `monitoring_items_i_chart_template.md` 中用例名称包含 `未分析的单值控制图样本不足`、`未分析的单值控制图字段仍可匹配`、`未分析的单值控制图纵轴变量缺失` 的用例写法；模板仅作参考，生成时按以下要求执行。

- **标题**：`未分析的<方法><具体未分析原因>,[修复后参与一键分析成功 | 一键分析成功 | 仍保持未分析]`，标题中必须出现具体原因（如样本不足、字段删减、变量类型变化、阶段未匹配等），不得只写笼统的“未分析修复后成功”。
- **前置条件**：必须写明“存在因<具体原因>处于未分析状态的<方法>”；修复成功类用例还需写明“已准备可替换的数据源或可执行的数据处理操作”。
- **步骤**：必须包含“进入数据分析详情页，确认<方法>为未分析状态且提示<具体原因>”这一步，再写触发数据替换或数据处理并确认一键分析的动作。
- **预期结果**：
  - 修复成功类：状态由未分析变为已分析；分析结果名称填充为本次分析的标题；分析图（及当前方法适用的过程数据、判异检查、超出参考线检查等结果区域）按修复后的数据生成；未分析标签和悬浮原因被清除。
  - 仍保持未分析类：状态保持未分析；悬浮"?"提示具体原因；配置区域涉及字段的处理按"未分析项修复判定"口径执行（字段清空时不显示 NULL）。

### 报告插入分析结果写法

本类用例可参考 `monitoring_items_i_chart_template.md` 中用例名称为 `单值控制图分析结果能正常插入报告，结果显示正确` 的用例写法；同类用例建议归入 `报告编制 / 分析结果插入` 分组，模板仅作参考，生成时按以下要求执行。

- **前置条件**：必须写明“已具备报告编辑权限”；“存在分析方法为<方法>且分析状态为已确认的监控项目”；以及“数据分析工作台存在已分析的<方法>结果”（监控项目和工作台两个数据来源都要写）。
- **步骤**：必须体现插入弹窗的两个来源 tab——“点击插入分析结果”后，“在监控项目 tab 页选择<方法>，在分析平台 tab 页选择<方法>”，再“确认插入并查看报告内容”；不得只写“插入分析结果”一笔带过。
- **预期结果**：写明“<方法>插入报告成功，分析图、过程数据、判异检查、超出参考线检查和输入结论显示正确”，按当前方法 PRD / UI 实际存在的结果区域逐项列出；不得只写“分析图和输入结论”等部分区域。

### 自定义结论写法

本类用例可参考 `monitoring_items_i_chart_template.md` 中用例名称包含 `自定义结论含中文、字母、小数、特殊字符时可保存成功`、`自定义结论最多支持 2000 个字符，可保存成功` 的用例写法；同类用例归入 `数据分析工作台 / 数据分析 / 自定义结论` 分组，模板仅作参考，生成时按以下要求执行。

- **字符类型**：用例名如“<方法>自定义结论含中文、字母、小数、特殊字符时可保存成功”；预期：可保存成功，重新进入详情页后回显与输入一致。
- **字符长度边界**：用例名如"<方法>自定义结论最多支持 2000 个字符，可保存成功"；预期：最大字符数 2000 可保存并完整回显，超过 2000 时不能输入或被拦截。字符长度上限固定为 2000，不以当前分析方法 PRD / UI 为准，无需纳入需求问题清单。
