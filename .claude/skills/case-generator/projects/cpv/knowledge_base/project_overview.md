# 项目整体介绍

## 项目背景

本系统主要服务于制药企业商业化生产阶段，覆盖原料药、制剂和生物制品等产品类型。系统围绕产品、工艺路线、CPP、CQA、CMA、IPC 和生产数据建立结构化数据模型，用于支撑持续工艺确认、趋势监控、统计分析、报告编制和合规追溯。

## 核心价值主张

| 价值主张 | 说明 |
|---|---|
| 确保安全 | 通过持续监控工艺表现，及时识别潜在质量风险，保障药品安全、有效和质量可控。 |
| 降低合规风险 | 支撑电子记录、电子签名、审计追踪和数据完整性要求，便于应对监管审计。 |
| 提升运营效率 | 自动化数据收集、分析和报告编制，减少人工整理与重复分析工作。 |
| 驱动持续改进 | 通过趋势分析、统计分析和监控结果识别工艺优化机会。 |
| 增强决策支持 | 为产品放行、变更控制、年度回顾和持续工艺确认提供数据依据。 |

CPV 系统是连接生产操作、质量管理和持续改进的数字化枢纽，目标是确保生产工艺在商业化生产阶段持续受控。

## 系统概述

CPV（持续工艺确认）系统是一个面向制药行业工艺验证的闭环管理平台。它通过年度计划驱动、任务执行跟踪、方案模板标准化、监控项目数据分析和报告编制，实现从计划、方案、分析到报告的线上化合规管理。

业务术语统一维护在 `business_glossary.md`，核心业务流程统一维护在 `core_flows/cpv_business_flow.md`。本文件维护项目整体定位和菜单结构，不重复维护术语解释和详细流程。

## 主要功能范围

CPV 系统按站点分为公共管理站点和自定义站点（业务站点）：

- 公共管理站点负责站点、用户、权限、系统服务、系统界面配置、审计追踪和登录日志等全局管理能力。
- 自定义站点承载 CPV 业务功能，包括产品工艺、计划、任务、方案、相关性分析、监控项目、数据管理与洞察、报告和业务站点配置等能力。

## 菜单结构索引

本索引用于统一 CPV 系统菜单层级和标准模块名。生成测试用例时，用于判断需求所属站点、菜单路径、模块命名和参考用例匹配。

维护规则：

- 菜单名称以系统界面实际显示为准。
- `标准模块名` 用于生成用例标题、输出文件名和跨文档术语统一。
- `三级菜单` 不存在或暂未明确时填写 `-`。
- 产品数据权限适用范围只在 `user_roles.md` 中维护，本文件不重复维护。
- 若菜单名称和历史参考用例索引不一致，在备注中记录别名或历史名称，不直接改业务标准名。
- `方案管理`、`报告管理` 作为界面一级菜单名保留；生成用例时以二级菜单和 `标准模块名` 识别具体业务模块，如 `方案编制`、`报告编制`。

## 公共管理站点与自定义站点关系

| 项目 | 关系说明 |
|---|---|
| 系统定位 | 公共管理站点负责全局基础数据和权限能力，自定义站点负责具体 CPV 业务数据和业务流程。 |
| 数据隔离 | 各自定义站点业务数据相互隔离；公共管理站点不处理 CPV 计划、任务、方案、监控项目或报告等业务数据。 |
| 数据同步 | 公共管理站点向自定义站点同步用户、站点和权限等基础数据；用户账号全局唯一。 |
| 页面跳转与站点切换 | 用户可在有权限的公共管理站点和自定义站点之间切换，无需重新登录。 |
| 统一认证 | 所有站点共用同一套用户认证体系，会话信息用于控制可访问站点、菜单和权限。 |
| 站点管理员权限 | 自定义站点管理员管理所属站点内的业务配置、本地用户与菜单权限；公共管理站点管理员管理全局站点、用户和权限能力。 |
| 数据分析平台 | 各自定义站点共用数据分析平台能力，但分析数据和结果按站点隔离。 |
| 公共管理站点特性 | 公共管理站点在系统初始化时自动创建，站点编号为 `PUBLIC_ADMIN`，站点名称为“公共管理”，不可删除、不可停用。 |
